# Push Notifications - Tuk Talk Thai

This document covers how notification delivery works, how to deploy it, and how to test it safely.

## Architecture

```text
React client / PWA
  -> OneSignal Web SDK
  -> stores OneSignal player ID in Supabase profiles

Supabase Database Webhooks
  -> POST milestone payloads to send-notification
  -> authenticate with X-Tuk-Notification-Secret

pg_cron hourly tick
  -> calls public.tick_notifications()
  -> POSTs { "mode": "tick" } to send-notification
  -> authenticates with X-Tuk-Notification-Secret from Vault

send-notification Edge Function
  -> validates X-Tuk-Notification-Secret for every POST
  -> checks user preferences and cooldowns
  -> sends via OneSignal REST API
```

## Notification Types

| Type | Trigger | When it fires |
|---|---|---|
| `daily_reminder` | pg_cron hourly tick | 1 hour before the user's typical study time, if they have not studied today |
| `streak_warning` | pg_cron hourly tick | 22:00 local time, if the user has a 3+ day streak and has not studied today |
| `milestone_mission` | Database Webhook on `user_missions` insert/update | When `completed=true` is written |
| `milestone_stage` | Database Webhook on `user_stats` update | When `current_stage` increments |
| `milestone_xp` | Database Webhook on `user_stats` update | When `total_xp` crosses 100, 500, 1000, or 5000 |
| `re_engagement` | pg_cron hourly tick | At typical study hour after 7+ inactive days |
| `new_content` | OneSignal dashboard | Manual broadcast by the owner |

The Edge Function checks each user's `notification_preferences` before sending. The `new_content` type is sent directly from OneSignal and is not generated by app code.

## One-Time Setup

### 1. Deploy the Edge Function

Supabase CLI:

```bash
supabase functions deploy send-notification --project-ref fkebzcywofzloaqeghtn --use-api --no-verify-jwt
```

Dashboard alternative:

1. Supabase Dashboard -> Edge Functions -> Create a new function.
2. Name: `send-notification`.
3. Paste `supabase/functions/send-notification/index.ts`.
4. Deploy with JWT verification disabled, because the function enforces its own POST secret header.

### 2. Set Edge Function secrets

Supabase Dashboard -> Edge Functions -> Manage secrets:

| Name | Value |
|---|---|
| `ONESIGNAL_APP_ID` | `9dff8341-a44c-4b22-a863-467baabd6f7d` |
| `ONESIGNAL_REST_API_KEY` | OneSignal REST API key; sensitive, never commit |
| `NOTIFICATION_WEBHOOK_SECRET` | Strong random value; sensitive, never commit |

`SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` are auto-injected by Supabase. Do not add the service-role key manually to docs, migrations, Vercel, webhook headers, or cron headers.

### 3. Configure the scheduler

1. Generate or retrieve the current `NOTIFICATION_WEBHOOK_SECRET` from a secret manager.
2. Open `supabase/migrations/004_notification_scheduler.sql`.
3. Replace `YOUR_NOTIFICATION_WEBHOOK_SECRET_HERE` with the actual value.
4. Run the SQL in Supabase SQL Editor or through an approved migration process.
5. Do not commit the edited SQL file with a real secret.

The scheduler setup:

- Enables `pg_net` and `pg_cron`.
- Stores the dedicated notification webhook secret in Vault as `notification_webhook_secret`.
- Creates `public.tick_notifications()`.
- Schedules `tuk-talk-notification-tick` at `5 * * * *`.

Verify the schedule:

```sql
select jobname, schedule, active
from cron.job
where jobname = 'tuk-talk-notification-tick';
```

### 4. Configure Database Webhooks

Supabase Dashboard -> Database -> Webhooks.

Webhook 1:

- Name: `mission_complete_notify`
- Table: `user_missions`
- Events: Insert and Update
- Type: HTTP Request
- Method: POST
- URL: `https://fkebzcywofzloaqeghtn.supabase.co/functions/v1/send-notification`
- Headers:
  - `Content-Type: application/json`
  - `X-Tuk-Notification-Secret: <NOTIFICATION_WEBHOOK_SECRET>`

Webhook 2:

- Name: `stats_update_notify`
- Table: `user_stats`
- Events: Update
- Type: HTTP Request
- Method: POST
- URL: `https://fkebzcywofzloaqeghtn.supabase.co/functions/v1/send-notification`
- Headers:
  - `Content-Type: application/json`
  - `X-Tuk-Notification-Secret: <NOTIFICATION_WEBHOOK_SECRET>`

Do not use service-role bearer authorization in either webhook.

## Verification

Health check:

```bash
curl https://fkebzcywofzloaqeghtn.supabase.co/functions/v1/send-notification
```

Expected:

```json
{"ok":true,"configured":true}
```

Unauthorized POST should fail:

```bash
curl -i -X POST \
  https://fkebzcywofzloaqeghtn.supabase.co/functions/v1/send-notification \
  -H "Content-Type: application/json" \
  -d '{"table":"noop","record":{}}'
```

Expected: HTTP `401`.

Authorized no-op webhook test:

```bash
curl -i -X POST \
  https://fkebzcywofzloaqeghtn.supabase.co/functions/v1/send-notification \
  -H "X-Tuk-Notification-Secret: <NOTIFICATION_WEBHOOK_SECRET>" \
  -H "Content-Type: application/json" \
  -d '{"table":"noop","record":{}}'
```

Expected: HTTP `200` with a skipped response.

Manual send test with a subscribed test user:

```bash
curl -X POST \
  https://fkebzcywofzloaqeghtn.supabase.co/functions/v1/send-notification \
  -H "X-Tuk-Notification-Secret: <NOTIFICATION_WEBHOOK_SECRET>" \
  -H "Content-Type: application/json" \
  -d '{"mode":"send","user_id":"USER_ID","type":"daily_reminder","data":{}}'
```

This can send a real notification. Use only with a controlled subscribed test account.

Metadata checks should show no service-role bearer auth in notification triggers or cron:

```sql
select
  t.tgname,
  position('Authorization' in pg_get_triggerdef(t.oid)) > 0 as contains_authorization,
  position('Bearer' in pg_get_triggerdef(t.oid)) > 0 as contains_bearer,
  position('X-Tuk-Notification-Secret' in pg_get_triggerdef(t.oid)) > 0 as contains_secret_header
from pg_trigger t
join pg_class c on c.oid = t.tgrelid
join pg_namespace n on n.oid = c.relnamespace
where n.nspname = 'public'
  and t.tgname in ('mission_complete_notify', 'stats_update_notify');
```

```sql
select
  position('service_role_key' in pg_get_functiondef(p.oid)) > 0 as references_service_role_vault_secret,
  position('Authorization' in pg_get_functiondef(p.oid)) > 0 as contains_authorization,
  position('Bearer' in pg_get_functiondef(p.oid)) > 0 as contains_bearer,
  position('X-Tuk-Notification-Secret' in pg_get_functiondef(p.oid)) > 0 as contains_secret_header
from pg_proc p
join pg_namespace n on n.oid = p.pronamespace
where n.nspname = 'public'
  and p.proname = 'tick_notifications';
```

Do not print raw trigger definitions or raw function bodies in shared logs, because webhook metadata contains the dedicated notification secret.

## Testing Checklist

- OneSignal SDK initializes after sign-in.
- Browser permission flow appears only after onboarding.
- Subscription ID is saved to `profiles.onesignal_player_id`.
- Timezone is saved to `profiles.timezone`.
- Edge Function health returns `configured: true`.
- Missing `X-Tuk-Notification-Secret` returns `401`.
- Valid `X-Tuk-Notification-Secret` reaches the no-op webhook path.
- `mission_complete_notify` uses the custom header and has no bearer auth.
- `stats_update_notify` uses the custom header and has no bearer auth.
- `tick_notifications()` uses the custom header and has no bearer auth.
- `tuk-talk-notification-tick` remains active on schedule `5 * * * *`.
- A controlled subscribed test user can receive a manual notification.
- Mission and stats milestone notifications work after owner test data changes.

## Troubleshooting

### Edge Function returned 401

The request is missing `X-Tuk-Notification-Secret`, or the value does not match the Edge Function secret.

### Health check returns `configured: false`

One or more required Edge Function env vars are missing:

- `ONESIGNAL_APP_ID`
- `ONESIGNAL_REST_API_KEY`
- `SUPABASE_URL`
- `SUPABASE_SERVICE_ROLE_KEY`
- `NOTIFICATION_WEBHOOK_SECRET`

### Cron job runs but no notifications send

Check Edge Function logs and OneSignal response status. Common causes:

- `ONESIGNAL_REST_API_KEY` is missing or invalid.
- No user has `onesignal_player_id` populated.
- Users are not in a matching local-hour notification window.
- User notification preferences disable the notification type.
- The cooldown window suppresses non-milestone notifications.

### OneSignal subscription is not saving

Check the browser console for OneSignal errors. Common causes:

- Browser notification permission is denied.
- iOS Safari user has not installed/opened the PWA.
- The OneSignal app domain does not match the live app origin.

### Notification opens the wrong URL

The Edge Function defaults to `https://thai-fluency.vercel.app/`. OneSignal dashboard broadcasts should use the same origin unless intentionally linking elsewhere.

## Security Notes

- The OneSignal REST API key lives only in Supabase Edge Function secrets.
- The notification webhook secret lives in Supabase Edge Function secrets, Supabase Vault for cron, and Database Webhook headers.
- The notification webhook secret is purpose-specific. Rotate it if it is printed, committed, or shared outside the owner/operator workflow.
- The Supabase service-role key is used only by the Supabase-managed Edge Function runtime. It must not be placed in Database Webhook headers, cron headers, Vault entries for notification dispatch, Vercel env vars, docs, or migrations.
- Vercel only needs public client env vars: Supabase URL, public client key, and OneSignal app ID.
- Users can disable notification categories in the profile notification settings.
- Users can revoke browser notification permission in browser/site settings.
